# Grounded-answer invariants

Standing rules for any surface that generates, displays, or advertises answers. These outlive any single build — current implementation spec: `docs/grounded-ask-spec.md`.

1. **Cite or refuse.** No generated answer ships without at least one citation resolving to an official source URL (bill → `Bill.official_url`, roll call → `VoteEvent.official_url`, legislator → `Legislator.profile_url`). Weak retrieval refuses rather than stretches. An honest "no matches" or "out of scope" is a first-class response, not an error state.

2. **Never advertise what you can't answer.** Placeholders, sample question chips, suggested follow-ups, empty states, and marketing copy may only name intents the router can currently answer. System-suggested questions (chips) must be constructed so they cannot lead to a refusal.

3. **Grounded neutrality.** Describe records, never inferred positions: "authored / co-authored N bills," "voted yes on HF xxxx" — never "supports X" as an opinion claim. Use Minnesota Legislature terminology: author / co-author (not "sponsor"/"co-sponsor") in user-facing copy. This governs *displayed strings only* — code identifiers and the data model keep their names, including the genuine `SponsorshipRole.sponsor` enum value, which is a distinct role from chief/co-author (its semantics are tracked in `docs/grounded-ask-spec.md` §4.2, Sponsorship display). Framing sentences that carry this rule are fixed UI copy owned by the layout, never LLM output.

4. **Records surfaces vs. generated answers.** Data surfaces (bill pages, Votes tab) render facts from the DB and may show anything the record contains. Generated answers carry the citation contract. Never leak record data (e.g., vote tallies) into a generated answer ahead of its cited answer path shipping.

5. **Anything linked to must be URL-addressable.** If a citation, CTA, or cross-page navigation targets a location (a tab, a highlighted passage, a roll-call row), that location must be reachable via URL (`?tab=votes`, `#anchor`) — component state cannot be linked from another page. Shareable URLs are a feature of every such surface, not an afterthought.

6. **Copy claims match shipped capability.** Product copy (hero subhead, deflection text, badges) asserts only what the shipped surfaces deliver — e.g., "how everyone voted" requires visible tallies + official-record links somewhere in the product. If a capability slips, trim the claim in the same release; never mock or ship copy around a gap.

7. **Answerable means ingested and fresh — keep the corpus current.** Every answer path is only as current as the ingested corpus: Grounded Ask can answer solely from what ingestion has actually pulled in, **at the state it was pulled in**. Any surface implying currency — the hero's "Minnesota law … where it stands," a "new law" framing, current-session chips — presumes the relevant session is in the corpus *and its records reflect reality*, so the **active / most-recent legislative session must be ingested and refreshed** before that copy ships (this is the currency face of rules 2 and 6). Staleness is the sneakier failure: ingestion is human-triggered and skips already-ingested bills by default, so a bill can sit in the corpus at a months-old status — production held HF 4138 at a pre-passage status two months after it was signed (found Jul 2026) — and a status-stale answer misframes enacted law as a pending proposal, violating the status-aware framing this product promises. Canonical current-law questions are **standing acceptance tests, not one-off examples**: "What's in the new social media law for kids?" must resolve to the enacted 2026 bill (HF 4138) with a cited answer whose framing says *law*, not *bill* ([#155](https://github.com/alethical-org/alethical/issues/155)). A persistent refusal on genuinely-enacted current law — or a confidently *stale* answer about it — is a **gap to close, not an acceptable steady state**: the refusal is still the correct behavior while a coverage gap exists (rule 1), but the gap is a bug against this rule, and it is the maintainer's job to notice a canonical use case has gone unanswerable or stale and fix the corpus, unprompted.

8. **Follow-up chat is a standing capability — answer-first, never router-opened.** Signed-in users can continue a single-bill answer conversationally: the bill page's chat entry keeps working, and the answer page's follow-up composer hands off into the existing bill-scoped chat seeded with the Ask question + generated answer (`docs/grounded-ask-spec.md` §9.2, Answer page shared elements — decided Jul 2026). The hero/router never opens a chat session directly: every Ask produces a one-shot answer page first, and chat is a continuation surface, not a routing destination (there is deliberately no `chat` intent in the router). Changes to any Ask surface must not regress the bill-scoped chat — the standing enforcement is the contract tests `test_signed_in_chat_session_and_message_flow` and `test_bill_scoped_chat_missing_chunks_returns_grounded_fallback` (`alethical/tests/test_api_contract.py`).
