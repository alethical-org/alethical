# Session workflow rules

Machine-facing counterparts of the human conventions in `CONTRIBUTING.md`. Follow both; when they seem to conflict, CONTRIBUTING.md wins.

1. **Target branch first.** Establish at the start which branch or PR the work belongs to; switch the worktree there and edit in place. Don't build on a session branch and port the files later. Before starting new work, scan the open PRs (`gh pr list`) for the same files or topic — with parallel sessions, duplicating in-flight work means the same change merges twice and conflicts once (#121/#122). If an open PR already covers it, build on that branch or wait for it.

   **`git fetch origin main` before you start building, and check *just-merged* PRs on your target file, not only open ones.** The session-start "N commits behind" snapshot goes stale the moment another session merges — origin advances mid-session. An overlap that already merged is invisible to an open-PR scan and surfaces only later, as a whole-file reformat or a merge conflict once you've built on a stale base. So before writing code: `git fetch origin main`, then `git log --oneline HEAD..origin/main -- <target file>` and `gh pr list --state merged --search "<file/topic>"`; if origin moved or a merged PR already touched the file, rebase and re-scope as an extension before you build (mobile ask-field, Jul 2026: #180 and #186 each merged mid-session and forced a rebuild).

2. **Commit and push at milestones, not at session end.** Uncommitted work is invisible — not in the PR, not reviewable, not recoverable by anyone outside the conversation. When a coherent chunk lands, commit it.

3. **Share branches, not file copies.** If the user hands you a loose copy of a repo file (Downloads, attachment), diff it against the branches before building on it — it may duplicate or trail a branch. Land durable content on a branch promptly, and point people at branches/PRs rather than exporting files. The moment a loose copy's content lands on a branch, delete the loose copy in the same session — a kept copy immediately starts trailing the branch and resurfaces later as a stale shadow of main.

4. **Route knowledge at birth.** Decisions → the relevant spec in `docs/`. Tasks and deferred work → GitHub issues, filed at the moment of discovery with: what it is · what exists today · why deferred · what unblocks it · **a milestone and an effort label** (`effort: small`/`medium`/`large`, per `CONTRIBUTING.md`) assigned at filing, not later (then cross-link the issue number back into the doc that spawned it). Product invariants → `.claude/rules/`. Team workflow norms → `CONTRIBUTING.md`. Chat is a scratchpad, never a system of record.

   The same applies to *existing* issues: when work you ship or a decision you record changes one — done, half-done, re-scoped, obsolete — update or close it in the same session. Monthly triage is the backstop, not the mechanism.

5. **Verify cheap repo facts before deciding on them.** If a claim is one `diff`, `git log`, or `grep` away from certain (which version is newer, does the model have that field, is the file gitignored), check it before building work or advice on top of it. Scope and timing claims are the flagship case: never assign MVP/post-MVP or milestone timing from memory — check `gh issue list`, the milestones, and `docs/v1-scope.md` first, and cite the issue, milestone, or spec section that grounds the claim.

6. **After a scope change, search for everything that still describes the old scope.** Rule 4 covers artifacts you already know are affected; this rule covers the ones you don't. Whenever a decision re-scopes, re-phases, renames, or draws a boundary (descoping a capability, adding to out-of-scope, changing a DoD, changing terminology), actively *search* for artifacts that still describe the old state — `gh issue list --search "<feature keywords>"` for the tracker, `grep` across `docs/` and `.claude/rules/` for specs and rules — and update what the search finds in the same session, unprompted. Record the result in the PR body ("Stale-reference check: #79 rescoped" / "Stale-reference check: nothing affected") so reviewers can see it happened. The trigger is *making* the change, not being asked about it.

   `grep` can't see inside binary assets, so screenshots and diagrams embedded in any doc (build specs, onboarding guides, READMEs — not just `*-spec.md`) are the stale reference easiest to miss: when a change alters what a doc's screenshot depicts (copy, layout, states, chips), refresh the embedded image in the same change so the doc's visual and text never diverge. A chat-attached image isn't a file on disk — locate the source (screenshots land in `~/Desktop/Screenshot/` as `.jpg`), verify it's the right frame, and convert to match the tracked asset's format (`sips -s format png`) so the path stays stable.

7. **Name things so a newcomer can guess what they are.** Branch names, PR titles, filenames, headings, and rule names describe the thing literally, in plain words: `docs/update-issues-on-scope-change`, not `docs/ripple-sweep-habit`. No metaphors, coined terms, or shorthand that only makes sense to whoever was in the conversation that produced it. Name for what the change *does or decides*, not for where it lands — the diff already shows the location. If a coined term genuinely earns its keep, define it in plain words at first use in the doc that owns it.

8. **Spell out spec references — file name plus section topic.** When citing a spec section in any artifact (docs, issues, PR bodies and comments, chat), write the full spec file name and what the referenced section is about — its header or topic: "`docs/grounded-ask-spec.md` §9 (Answer page UI — v1 states)", never "the spec §9" or a bare "§9.4". Someone new reading the sentence in isolation should know exactly what's being referred to without opening anything. Once the full form has appeared in an artifact, later mentions there may shorten. The same principle applies to every pointer: issues and PRs get markdown links with their titles or a gloss, never a bare number.

9. **Resolve factual claims from the primary source, yourself.** When a fact that will be shown to users or baked into product content, copy, or a mockup can be confirmed from a primary source — a bill's enacted text, a statute section, an official roll-call record, an authoritative page — fetch and read that source and confirm the exact value (effective dates, verbatim quotes, section/subdivision numbers, vote tallies) before asserting it. Don't relay a secondary summary, trust a skim, or hand over an unverified value with "please confirm" — do the digging and resolve it. When sources conflict (e.g. a status-page metadata field shows one effective date but the enacted bill's EFFECTIVE DATE clause shows another), the authoritative primary text wins; state the discrepancy and which source you trusted. This extends rule 5 from repo facts to external primary sources, and it matters most for anything user-facing, where a confident-but-wrong cited fact is the worst failure (`.claude/rules/grounded-answers.md`).

10. **Just do safe, reversible work — don't propose-and-offer.** When a change is safe, reversible, and outside app / backend / production code — docs, specs, `.claude/rules/`, `.claude/` config, `.gitignore`, branch hygiene, cross-links, filing or updating issues — carry it out and report the result; don't narrate it and ask whether to proceed. That includes the whole loop: branch, commit, open the PR, squash-merge it, then clean up so nothing lingers — `--delete-branch` removes the remote, and **reset your local branch to `origin/main`** (`git fetch origin main && git reset --hard origin/main`) so it isn't left pointing above main (a merged branch whose local pointer sits above main resurfaces as a stale "Create PR" prompt for the user). Rules 1–2 still apply — right target branch, scan open PRs first. The test is blast radius and reversibility, not size: if `git revert` or deleting a file cleanly undoes it and nothing user-facing ships, act; if it touches app/backend/production code, changes hard-to-reverse state (schema/data migrations, deletions, access/permission changes), or is outward-facing, propose first and get an explicit yes. This is the referent for `coding-discipline.md` rule 1's "those autonomy norms stand."

   **Safe, reversible front-end production changes are auto-eligible too** — not only presentational ones. This includes **implementing a design handoff** (building or reworking a screen or component from a mockup, spec, or design instructions — the `implementing-design-handoffs` case), styling, spacing, hover/focus/tap/active states, transitions and animations, responsive reflow, shared style-helper extraction, **accessibility / WCAG-2.x-AA fixes to already-shipped UI** (contrast, accessible names/labels, heading & landmark semantics, visible focus, keyboard operability, `prefers-reduced-motion`, role/`aria` mappings), and **self-contained client-side interaction logic** (auto-growing/expanding/collapsing inputs, local or transient UI state, keyboard handling, client-side sort/filter of already-loaded data). It qualifies only when the change is self-contained to its screen/component and touches **no** data-fetching/API/backend contract, auth, persisted or shared state, or schema, and makes **no** new user-facing claim or capability; `tsc --noEmit`, `prettier --check .`, and the production build pass; you have **manually verified it on the dev server across the affected viewports and states** (no front-end test runner yet — [#173](https://github.com/alethical-org/alethical/issues/173) — so manual QA is the gate); and a clean `git revert` fully undoes it. Say in the PR what you verified. **Accessibility fixes verify by their standard, not by eye, and are not held for approval:** file the issue at discovery, fix, then gate on `tsc`/`prettier`/CI/build green **plus** a WCAG check (screen-reader name/heading, reduced-motion toggle, a measured contrast ratio) — for an already-shipped public surface, live confirmation may follow the merge; when an AA fix darkens a shared token or carries another visual tradeoff, still merge but name the tradeoff in the after-report so it can be eyeballed and reverted if disliked. **Local routing wiring** qualifies only if you've confirmed every affected route — and any shareable/deep-link URL (`.claude/rules/grounded-answers.md` rule 5) — still resolves. Still propose-first: data / auth / persisted-state / schema / migrations, new-capability or claim-making copy (`.claude/rules/grounded-answers.md` + copy-claims invariants), outward-facing / published content, and anything you can't visually verify (e.g. an auth-gated surface). The governing test is **blast radius + reversibility + verifiability** — not whether it's "just styling" or "production code." Bias toward acting; keep widening this carve-out as new safe patterns prove out (propose the widening here in the same session), and propose narrowing it if something bites.

   **Safe, verified backend changes are auto-eligible too (granted Jul 14 2026).** A backend / pipeline / API change qualifies for build-and-squash-merge when **all** of these hold: it is behavior-preserving by default or its new behavior is fully exercised by tests (e.g. a parameterization whose defaults keep current behavior, a bug fix with a regression test); the full backend suite passes **locally against real Postgres** and the **backend CI job is green** on the PR; a clean `git revert` undoes it (no schema/data migration, no destructive or irreversible state change); and it spends no money and touches no production *data* (writing code that will later run against production is fine — *running* it against production is not). Reference precedents: [#219](https://github.com/alethical-org/alethical/pull/219) (session parameterization, defaults unchanged) and [#222](https://github.com/alethical-org/alethical/pull/222) (fallback-embedding labeling fix with regression test + real-data proof) — both CI-green and merged under this clause's grant. Say in the PR what was verified and how. **Still propose-first, always:** schema / data migrations (Alembic), anything auth/security-sensitive, destructive or hard-to-reverse data operations, production data runs (backfills, ingests, enrichment — these also usually spend money), new externally-visible API contracts or user-facing claims, and changes whose correctness you cannot demonstrate by test or local execution. The governing test stays blast radius + reversibility + verifiability; "it's backend code" alone is no longer a reason to hand the merge back.

   **Zero-behavior cleanup is always in-lane — correct it, don't offer.** A change that alters no runtime behavior and makes no new user-facing claim clears the carve-out *by construction*, so just do it and merge: fixing a stale, orphaned, or now-false code comment or reference that a change created (rule 6 fallout), a typo, a dead-comment tidy-up, a broken cross-link, or the like. This holds **even in app / front-end code, and even in code another session just shipped** — correct it to match current reality now; never defer on "it's their code," "the underlying value might change later," or "the user is wrapping up." Verify the cheap way (the diff itself for prose; `tsc` / `prettier` when the file also compiles), then PR and squash-merge in the same session. Catching yourself about to ask "want me to fix this comment / tidy this up?" *is* the propose-and-offer this rule forbids — the answer is already yes. (Still propose-first only if the "cleanup" quietly changes behavior, a claim, or a public contract — then it isn't zero-behavior and a different clause governs.)

   **A change you recommend is as authorized as one the user typed — don't hand the commit back.** The autonomy test is blast-radius + reversibility + verifiability, never *who thought of it*. The previous clause covers zero-behavior cleanup; this covers the rest: a change that *does* alter behavior but is still safe / reversible / verifiable and came from your own analysis — a follow-on you spotted, a **ripple from another session's merge** (Jul 2026: [#206](https://github.com/alethical-org/alethical/pull/206) shortened the capability cards' press-glow to 300ms, leaving the chips at 650ms — inconsistent with what [#204](https://github.com/alethical-org/alethical/pull/204) built, so aligning them ([#208](https://github.com/alethical-org/alethical/pull/208)) was mine to ship, not to hand back) — gets executed and reported after, exactly as if the user had asked. None of these turn it back into a "want me to ship it?": *it was my idea, not the user's* (the recommendation **is** the request — this makes binding, for the repo PR pipeline, `~/.claude/CLAUDE.md`'s "act on your recommendation, report after"), *it depends on inferring intent* (when the intent follows from the task's stated goal or the code, reason it out and act), or *another session chose the thing I'm rippling from deliberately* (deliberate-there ≠ off-limits-here). Reserve asking for the two genuine cases only: a real **toss-up** — no basis to prefer an option, so surface it briefly — and the **propose-first categories** (data / auth / persisted-state / schema / migrations, claim-making copy, outward-facing / published content, anything you can't verify). If the reason you're about to ask reduces to "it wasn't explicitly requested," that reason is void.

   **Applying `workflow-overhead-audit` verdicts is in this lane too.** When that audit produces keep/trim/merge/drop verdicts on non-exempt gates (rules, skills, docs, `.claude/` config), apply them in the same pass — including meaning-preserving rewrites of a rule or skill file. The audit record's before/after diff *is* the verification and `git revert` cleanly undoes it, so the "can't cheaply verify → propose-first" bar does not gate audit edits; don't stop to ask. Only the audit's own exempt safety rails stay off-limits (the leak-guard commit hook, `.claude/rules/grounded-answers.md` invariants, CI correctness/format checks). If you catch yourself about to ask "want me to merge this?" for a verified self-contained front-end change or an audit-verdict edit, the answer is already yes. The one thing that actually defeats this rule is reading a *stale* copy of it — a worktree branch or CLAUDE.md snapshot cut before a carve-out merged still shows the old "app code → propose first" text. Before deciding this rule doesn't cover your change, confirm you're on the current `origin/main` version (rule 1), not a stale checkout.

   **Execute the loop safely when sessions run concurrently.** The autonomous loop compresses build→merge into one unattended burst, and this repo runs many sessions and worktrees at once — origin, and sometimes your own working directory, move under you. Guard the merge, not just the start (rule 1):

   - **Isolate; never commit on a shared checkout.** If your working directory has been switched to another session's branch, or parallel worktrees are live, do the work in a dedicated worktree off `origin/main` (`git worktree add -b <branch> <path> origin/main`) and `git worktree remove` it when done. Don't switch the shared checkout's branch (it yanks the rug from the session using it) or build on whatever branch you happened to land on.
   - **Re-verify freshness at merge, against the fresh tip.** Right before committing and again before merging, `git fetch origin main` then `git diff origin/main -- <file>` — trust that, not your in-memory read: a file can mutate between your read and your edit when a PR merges mid-session (this session, Jul 2026: #200 rewrote a skill I had open; #203 landed while working). Confirm the diff is *only* your hunks — a tangled or duplicated diff means another session already shipped part of it (rule 1's #121/#122 trap, caught at merge instead of after).
   - **"Skipped" is not "verified."** A CLEAN merge state with the relevant CI job SKIPPED (path-filtered — e.g. the front-end job on a docs-only PR, so repo-wide `prettier --check .` never ran) means that gate didn't fire. You are the last gate in this lane: run the skipped check locally, or consciously accept its absence and say so in the PR.
   - **Finish cleanup by hand when a worktree holds `main`.** `gh pr merge --delete-branch` can't check out `main` or delete your local branch when another worktree holds them — its `failed to run git` prints *after* the remote squash lands, not instead of it. Confirm the squash is on the `origin/main` tip, then clean up by hand (`git branch -D <branch>`, `git worktree remove <path>`); don't read the error as a failed merge.
