# Session workflow rules

Machine-facing counterparts of the human conventions in `CONTRIBUTING.md`. Follow both; when they seem to conflict, CONTRIBUTING.md wins.

1. **Target branch first.** Establish at the start which branch or PR the work belongs to; work in your own worktree off `origin/main` for it (`just worktree <branch>` — see rule 10, concurrent-session isolation) rather than building in the shared checkout. Don't build on a session branch and port the files later. Before starting new work, scan the open PRs (`gh pr list`) for the same files or topic — with parallel sessions, duplicating in-flight work means the same change merges twice and conflicts once (#121/#122). If an open PR already covers it, build on that branch or wait for it.

   **`git fetch origin main` before you start building, and check *just-merged* PRs on your target file, not only open ones.** The session-start "N commits behind" snapshot goes stale the moment another session merges — origin advances mid-session. An overlap that already merged is invisible to an open-PR scan and surfaces only later, as a whole-file reformat or a merge conflict once you've built on a stale base. So before writing code: `git fetch origin main`, then `git log --oneline HEAD..origin/main -- <target file>` and `gh pr list --state merged --search "<file/topic>"`; if origin moved or a merged PR already touched the file, rebase and re-scope as an extension before you build (mobile ask-field, Jul 2026: #180 and #186 each merged mid-session and forced a rebuild).

2. **Commit and push at milestones, not at session end.** Uncommitted work is invisible — not in the PR, not reviewable, not recoverable by anyone outside the conversation. When a coherent chunk lands, commit it.

3. **Share branches, not file copies.** If the user hands you a loose copy of a repo file (Downloads, attachment), diff it against the branches before building on it — it may duplicate or trail a branch. Land durable content on a branch promptly, and point people at branches/PRs rather than exporting files. The moment a loose copy's content lands on a branch, delete the loose copy in the same session — a kept copy immediately starts trailing the branch and resurfaces later as a stale shadow of main.

4. **Route knowledge at birth.** Decisions → the relevant spec in `docs/`. Tasks and deferred work → GitHub issues, filed at the moment of discovery with: what it is · what exists today · why deferred · what unblocks it · **a milestone, an issue type, a topic label, and its board fields (Epic, Priority, LOE, Status)** (see the categorization rule just below) assigned at filing, not later (then cross-link the issue number back into the doc that spawned it). Product invariants → `.claude/rules/`. Team workflow norms → `CONTRIBUTING.md`. Chat is a scratchpad, never a system of record.

   The same applies to *existing* issues: when work you ship or a decision you record changes one — done, half-done, re-scoped, obsolete — update or close it in the same session. Monthly triage is the backstop, not the mechanism.

   **Categorize every issue at creation — the Roadmap board (org `alethical-org` Project #1 "Alethical Roadmap") is the source of truth.** Each question has exactly one home (the July 2026 consolidation retired the old `v0`/`post-v0` scope labels, the `bug`/`enhancement`/`documentation`/`question` kind-labels, the `data`/`ops`/`chat` topic-labels, and the `effort:` labels):
   - **Type** (native Issue Type: `Bug` / `Feature` / `Task`) — *what kind of work.* A docs task = `Task` + `documentation` label, not a separate type. Org-level, single-select, does not apply to PRs.
   - **Milestone** — *which release/phase.* `v0 hardening` → `v1` → `v2` → `v8 elections` → `v9 tbd` (the numeric `v#` prefix forces the list order; defer work by **moving an issue to a higher-numbered bucket**, never by renaming a milestone). Single source of truth for phase — no fractional milestones.
   - **Epic** (project single-select) — *which product area.* Grounded Ask & AI · Bills · Legislators · Votes & roll-calls · Candidates & Elections · Data & Ingestion · Platform & Ops · Tracking & Notifications · Home & Navigation · Notes & Pending (last = parked zero-risk items). Platform items also take a **Platform Track** (CI & Testing / Backend architecture / Deploy & Ops / Security & auth).
   - **Priority** (project single-select: Urgent / High / Medium / Low) — *how important*, **decoupled from milestone** (a High item sitting in v2 = a pull-forward candidate; a Low item in v0 hardening = must-do-soon but unimportant).
   - **LOE** (project **number** field) — *how big*, in points (small ≈ 1, medium ≈ 3, large ≈ 8) so views can **sum** it per epic/milestone. Parent epics carry **no** LOE (children only), so roll-ups don't double-count.
   - **Topic label** — *which stack layer.* `frontend` / `backend` / `documentation` / `auth` (cross-cutting facets; product areas are Epics now). Applies to PRs too.
   - **`research` label** — *work-character marker* (additive, not a stack layer): apply when the issue's main job is investigation — a spike, an eval, empirical tuning, or resolving an unknown to pick an approach before building (incl. agent-persona research). It keeps its Type and Epic; the label just flags the character so research/spike work is filterable.
   - **Status** (project Kanban) — *where it is in flight.* Backlog → Ready → In progress → In review → Done.

   Labels go in the `gh issue create` / `gh pr create` `--label` call; Type, Epic, Priority, LOE, and Status are native/project fields set via the Projects GraphQL API (`updateProjectV2ItemFieldValue`, `addProjectV2ItemById`, `addSubIssue`) — query the project's field/option IDs with `gh api graphql` (needs the `project` token scope). An uncategorized issue is the exception to fix, never the norm.

   **Keep the board current through the work lifecycle — every session, not just at creation.** The board is only useful if it reflects reality, so whenever a session **starts, changes, finishes, or discovers** work, update the item the same session: move **Status** (work started → In progress; PR open → In review; merged → Done) and **close the issue** when its work ships (`Closes #n` in the PR, or manually — closed issues auto-archive off the board); **add new issues to the board** fully categorized and **nested under their parent epic** when part of one (enable the project's *Auto-add* workflow so new issues land in Backlog automatically; until then, add them by hand); and re-set **Milestone / Epic / Priority / LOE / Type / labels** whenever the work changes the answer. On any scope change, search for and fix the stale issues and descriptions it affects (rule 6) — including the **milestone and epic descriptions themselves**, which encode the phase/boundary rules (e.g. what belongs in each epic), so keep them current as scope evolves. Leaving the board trailing reality is the exception to fix, never the norm.

   **Categorizing a new issue is two steps — and any session sweeps for ones that missed step two.** `gh issue create` sets only *repo* fields (`--label`, `--milestone`); the **board fields (Epic, Priority, LOE, Status) and the native Type are NOT set by it** and need a follow-up (Projects GraphQL or the board UI). An issue that lands in **"No Epic"** or with a blank Priority/LOE/Type is the tell that step two was skipped — this is how #416 slipped in (created with just a `backend` label). So **(a)** categorize fully at creation, and **(b)** as a self-healing backstop — don't trust that the creating session did it — whenever you start a session or touch the board, run the sweep and fix whatever it lists:
   ```
   gh project item-list 1 --owner alethical-org --format json --limit 200 \
     | jq -r '.items[] | select(.content.type=="Issue")
         | select(.epic==null or .status==null
                  or (.priority==null and .epic!="Notes & Pending")
                  or (.milestone==null and .epic!="Notes & Pending")
                  or (.lOE==null and (.content.title|startswith("Epic:")|not)))
         | "#\(.content.number) \(.content.title)"'
   ```
   The exclusions are by design: Notes & Pending items are priority/milestone-less, and `Epic:` parent issues carry no LOE (so roll-ups don't double-count). Type isn't in the project item-list, so also spot-check it (`gh issue list --json number,issueType`). Enable the project's **Auto-add** and **"Item added → Status: Backlog"** workflows so new issues at least reach the board in Backlog automatically; the judgment fields (Epic/Priority/LOE/Milestone/Type) still require a session to set.

5. **Verify cheap repo facts before deciding on them.** If a claim is one `diff`, `git log`, or `grep` away from certain (which version is newer, does the model have that field, is the file gitignored), check it before building work or advice on top of it. Scope and timing claims are the flagship case: never assign MVP/post-MVP or milestone timing from memory — check `gh issue list`, the milestones, and `docs/v1-scope.md` first, and cite the issue, milestone, or spec section that grounds the claim.

6. **After a scope change, search for everything that still describes the old scope.** Rule 4 covers artifacts you already know are affected; this rule covers the ones you don't. Whenever a decision re-scopes, re-phases, renames, or draws a boundary (descoping a capability, adding to out-of-scope, changing a DoD, changing terminology), actively *search* for artifacts that still describe the old state — `gh issue list --search "<feature keywords>"` for the tracker, `grep` across `docs/` and `.claude/rules/` for specs and rules — and update what the search finds in the same session, unprompted. Record the result in the PR body ("Stale-reference check: #79 rescoped" / "Stale-reference check: nothing affected") so reviewers can see it happened. The trigger is *making* the change, not being asked about it.

   `grep` can't see inside binary assets, so screenshots and diagrams embedded in any doc (build specs, onboarding guides, READMEs — not just `*-spec.md`) are the stale reference easiest to miss: when a change alters what a doc's screenshot depicts (copy, layout, states, chips), refresh the embedded image in the same change so the doc's visual and text never diverge. A chat-attached image isn't a file on disk — locate the source (screenshots land in `~/Desktop/Screenshot/` as `.jpg`), verify it's the right frame, and convert to match the tracked asset's format (`sips -s format png`) so the path stays stable.

7. **Name things so a newcomer can guess what they are.** Branch names, PR titles, filenames, headings, and rule names describe the thing literally, in plain words: `docs/update-issues-on-scope-change`, not `docs/ripple-sweep-habit`. No metaphors, coined terms, or shorthand that only makes sense to whoever was in the conversation that produced it. Name for what the change *does or decides*, not for where it lands — the diff already shows the location. If a coined term genuinely earns its keep, define it in plain words at first use in the doc that owns it.

8. **Spell out spec references — file name plus section topic.** When citing a spec section in any artifact (docs, issues, PR bodies and comments, chat), write the full spec file name and what the referenced section is about — its header or topic: "`docs/grounded-ask-spec.md` §9 (Answer page UI — v1 states)", never "the spec §9" or a bare "§9.4". Someone new reading the sentence in isolation should know exactly what's being referred to without opening anything. Once the full form has appeared in an artifact, later mentions there may shorten. The same principle applies to every pointer: issues and PRs get markdown links with their titles or a gloss, never a bare number.

9. **Resolve factual claims from the primary source, yourself.** When a fact that will be shown to users or baked into product content, copy, or a mockup can be confirmed from a primary source — a bill's enacted text, a statute section, an official roll-call record, an authoritative page — fetch and read that source and confirm the exact value (effective dates, verbatim quotes, section/subdivision numbers, vote tallies) before asserting it. Don't relay a secondary summary, trust a skim, or hand over an unverified value with "please confirm" — do the digging and resolve it. When sources conflict (e.g. a status-page metadata field shows one effective date but the enacted bill's EFFECTIVE DATE clause shows another), the authoritative primary text wins; state the discrepancy and which source you trusted. This extends rule 5 from repo facts to external primary sources, and it matters most for anything user-facing, where a confident-but-wrong cited fact is the worst failure (`.claude/rules/grounded-answers.md`).

10. **Just do safe, reversible work — don't propose-and-offer.** When a change is safe, reversible, and outside app / backend / production code — docs, specs, `.claude/rules/`, `.claude/` config, `.gitignore`, branch hygiene, cross-links, filing or updating issues — carry it out and report the result; don't narrate it and ask whether to proceed. That includes the whole loop: branch, commit, open the PR, squash-merge it, then clean up so nothing lingers — `--delete-branch` removes the remote, and **reset your local branch to `origin/main`** (`git fetch origin main && git reset --hard origin/main`) so it isn't left pointing above main (a merged branch whose local pointer sits above main resurfaces as a stale "Create PR" prompt for the user). Rules 1–2 still apply — right target branch, scan open PRs first. The test is blast radius and reversibility, not size: if `git revert` or deleting a file cleanly undoes it and nothing user-facing ships, act; if it touches app/backend/production code, changes hard-to-reverse state (schema/data migrations, deletions, access/permission changes), or is outward-facing, propose first and get an explicit yes. This is the referent for `coding-discipline.md` rule 1's "those autonomy norms stand."

   **Standing default — merge your own low-risk PRs yourself, then summarize (Eugene, Jul 14 2026).** This is the umbrella over every specific carve-out below (front-end, backend, docs, zero-behavior, self-recommended, audit-verdict). Any PR you author that is **low-risk — CI-green, cleanly `git revert`-able, its behavior verified (tests + dev-server QA as applicable), degrading safely, and clear of the propose-first categories** — you merge and report what it did; don't hand it back with a "want me to merge?" The bar is low-risk, not which lane it happens to fall in. **Propose-first still holds for genuinely higher-risk PRs:** *destructive/irreversible* schema / data migrations (an **additive, reversible, verified** migration is auto-merge — see the additive-migration clause below), auth / security *boundaries* (a *protective* feature like rate limiting is not one), destructive or hard-to-reverse state, production data runs that spend money, outward-facing *published* content, and copy asserting a capability the product can't yet back — and the top-level safety rules always gate regardless. When you're unsure a PR clears "low-risk," that doubt is itself the signal to ask. **Concurrency:** merging one PR can force an overlapping open PR to rebase + resolve before it merges (this session: #233 merging forced #230 to rebase) — resolve, re-verify green, then merge.

   **Standing autonomy grant — act on your recommendation without proposing first, across all work (Eugene, Jul 14 2026).** The maintainer removed the propose-first gate entirely: for every category the clauses above still marked "propose-first" — schema / data migrations, auth / security boundaries, outbound communications (email/notifications), production-data runs, outward-facing content, capability copy — build, verify, and merge on your own recommendation, proactively, without stopping to ask. "It's propose-first" is **no longer** a reason to pause; where any earlier or later clause in these rules says "still propose-first," this supersedes it. The autonomy test collapses to: **act, then report.** The **one** remaining hard line is the top-level safety obligation against *unverified irreversible real-world harm* — never let a build *fire* an irreversible external side effect unverified: a real send to real users, spending real money, or a destructive/irreversible production data / schema / access change. That line is honored by **engineering, not by asking**: build the capability fully, keep the live/irreversible trigger behind config (env/flag/dry-run default off), and verify with tests + a dry run, so the work ships autonomously without an unverified live blast. Capability copy still must match shipped capability (`.claude/rules/grounded-answers.md` rules 6–7) — that's a correctness constraint on the copy, not a reason to pause the build. Reserve a pause only for a genuine **toss-up** (no basis to prefer an option); everything else, proceed and report after.

   **An additive, reversible, verified migration is auto-merge — merge it yourself; auto-migrate-on-merge is the normal deploy path, not a gate (Eugene, Jul 21 2026).** This nails down the "schema / migrations" case of the grant above, because it's the one most easily mis-read back into propose-first. A schema migration that is **additive and reversible** — `CREATE TABLE`, add a nullable column, add an index, a new enum value — and that you have **verified** (an upgrade→downgrade→upgrade round-trip against real Postgres, plus CI green on the PR) is **in the auto-merge lane: build it, verify it, and squash-merge it on your own recommendation.** It is **not** the "destructive/irreversible schema change" the hard line reserves — that hard line is for genuinely *irreversible* migrations (`DROP COLUMN`/`DROP TABLE`, a data-losing type narrowing, a destructive `ALTER`), where `git revert` + downgrade would *not* cleanly restore state. Three traps this closes, all of which cost a needless handback this session (#489/#486 — an additive `CREATE TABLE` + additive, idempotent, dry-run-proven backfill that I built CI-green and then *asked about instead of merging*):
   - **`migrate.yml` auto-running `alembic upgrade head` against production on merge is the deploy pipeline, not a reason to gate.** Merging *is* how a verified migration reaches prod; that's the intended path. What makes merging safe is the **verification you already did** (the reversible round-trip + CI green), not a human nod. Don't reclassify "merge triggers the prod migration" as "merge fires an unverified irreversible blast" — for an additive/reversible/verified migration it fires a *verified, reversible* one.
   - **An additive, idempotent production backfill is likewise not the hard line** — its irreversible-harm guard is honored by **engineering** (dry-run default, then a scoped single-record live check read back), exactly as the grant says. Once those pass, run the full backfill and merge; don't hand it back.
   - **A task-embedded "propose-first" phrasing does NOT reinstate a gate the standing grant lifted.** A line like "any migration is propose-first via Alembic" inside a task prompt is subordinate to this standing grant; treat it as *"verify the migration especially carefully,"* not *"stop and ask."* Only a genuinely destructive/irreversible migration, or the maintainer **restating the gate as a live instruction in the current session**, actually pauses you. When in doubt whether a migration is reversible, that doubt is answered by *testing the downgrade*, not by asking.

   **Paid production runs are authorized when a zero-cost path isn't possible — spend from the budget to lead the work (Eugene, Jul 15 2026).** This refines the "spending real money" clause of the hard line above: money spend is **no longer an automatic gate** (wherever an earlier or later clause in these rules lists "money" / "spend" / "production-data runs that spend money" as a reason to pause, this supersedes that — cost alone never gates; verification of irreversible effects still does). The order of operations is (1) **prefer a zero-cost / near-zero path and prove it doesn't exist or won't do the job** before spending — investigate whether the outcome can be reached without paid API/LLM calls or a paid data run (e.g. re-deriving from data already present, re-fetching a *free* public source, a surgical column-only backfill instead of a full paid re-ingest); (2) if a zero-cost path suffices, take it; (3) if it genuinely can't, **run the paid production job on your own recommendation, funding it from the financial budget** — don't stop to ask merely because it costs money. The residual hard line is unchanged and is about *unverified irreversible harm*, not cost: before a run that spends real money **or** mutates/destroys real production data or hits real users, verify by engineering first — a **dry run + a small scoped live check** (one record end-to-end, read back, confirm correct) — then let the full run proceed. So: verify before you fire, prefer cheap over paid, but a correct, verified, budgeted paid run is authorized without a pause. (Origin: #328 action-date backfill — a zero-cost path existed, re-fetching the *free* MN Revisor XML and updating only the date columns, so no paid re-ingest was needed; had it not, this clause authorizes the paid re-ingest against the budget.)

   **Prioritize continuous, uninterrupted progress — don't stop for a next-step nod (Eugene, Jul 14 2026).** When you finish a unit of work, act on your own recommendation for the next best **unblocked** step and keep going; don't hand the decision back with "want me to do X, or hold?" Progress must not depend on the user being present to answer. **Not valid reasons to stop:** it's a big build, it's polish rather than critical-path, the critical path is gated on another session, or it wasn't explicitly requested — pick the next-best unblocked work and do it. When you hit a genuine blocker, surface it in one line **and immediately switch to the next-best unblocked thing** rather than idling. The only real pauses stay the **hard line** — an *unverified* irreversible real-world change (a destructive/irreversible prod data/schema/access change or a paid/live run you haven't verified; note an *additive, reversible, verified* migration or backfill is NOT this — merge it, per the clause above) and capability-overstating or outward-facing published copy — and a genuine **toss-up** with no basis to prefer.

   **Never offer a safe deliverable and wait to be asked — produce it (Eugene, Jul 15 2026).** Ending a turn with "say the word and I'll do X" / "want me to propose that?" / "I can draft that if you like" — for an X that is itself safe and reversible (a proposal, a table, an analysis, an options write-up, a plan, a doc/spec/rule/skill edit, a low-risk PR) — is the same handback this rule forbids, just disguised as politeness. **The recommendation *is* the trigger: if you'd recommend doing X next and X is in the safe/reversible lane, do X in the same turn and report it — don't surface X as an offer.** A *proposal itself is a safe output*: when the next step is "propose/table/analyze something," generate the proposal now rather than offering to generate it. This holds even when your recommendation is "no change" — state the decisive call and why, don't ask whether to look into it. The trigger to *withhold* and ask stays exactly the propose-first categories and a genuine toss-up; "the user might have a naming/taste preference" is not a toss-up when you have a clear recommendation — make the reversible call, report it, and note how to flip it. (Origin: after grouping the design skills I closed with "Say the word and I'll propose that table" for the non-design skills — a safe proposal I should have just produced.) **This explicitly covers recommended tweaks, hardening, and follow-up improvements you surface (Eugene, Jul 21 2026):** when you identify a safe/reversible/verifiable improvement — a CI guard, a config/workflow hardening, a refactor, a cleanup, a follow-up fix — *build and ship it in the same turn*; do not close with "want me to add the hardening? / say the word and I'll implement it." The fact that *you* proposed the tweak (rather than the user) is never a reason to gate it — the recommendation is the go. Only the propose-first categories and a genuine toss-up withhold. (Origin: after fixing #513 I closed with "Want me to (a) close it and (b) implement the hardening? Say the word" — I should have closed it and built the CI migration-collision guard [#523] autonomously, which is exactly what I then did once told to. Reserve asking for the real gates, not for tweaks you already recommend.)

   **Own the sequencing — decide the order of work yourself and drive the whole list to done (Eugene, Jul 14 2026).** When a task names a goal or an area but not a step list ("work on what makes sense next for X"), don't ask which item to start with — that decision is yours. Enumerate the sensible, in-scope items yourself, order them (unblocked-and-highest-value first), and work through them **continuously** with the goal of finishing all of them, not just the first. This is the sequencing counterpart to the continuous-progress default above and the standing autonomy grant: those say don't stop *between* steps and act-then-report; this says you also choose *what the steps are and in what order*. Consistent with that grant, the only genuine pause is a real **toss-up** (no basis to prefer). A per-item external dependency is **not** a pause: if an item needs a paid provider key, a finalized design handoff, or would otherwise fire an unverified irreversible side effect, build it behind config (env/flag, default off) and/or note it as "blocked on <what's needed>", then immediately move to the next item you *can* finish — don't let one gated item stall the list. Report the sequence and each item's outcome (shipped / built-behind-flag / blocked-on-what) at the end so the user sees the whole plan and where it stands, without having been asked to pick the order.

   **Safe, reversible front-end production changes are auto-eligible too** — not only presentational ones. This includes **implementing a design handoff** (building or reworking a screen or component from a mockup, spec, or design instructions — the `design-build` case), styling, spacing, hover/focus/tap/active states, transitions and animations, responsive reflow, shared style-helper extraction, **accessibility / WCAG-2.x-AA fixes to already-shipped UI** (contrast, accessible names/labels, heading & landmark semantics, visible focus, keyboard operability, `prefers-reduced-motion`, role/`aria` mappings), and **self-contained client-side interaction logic** (auto-growing/expanding/collapsing inputs, local or transient UI state, keyboard handling, client-side sort/filter of already-loaded data). It qualifies only when the change is self-contained to its screen/component and touches **no** data-fetching/API/backend contract, auth, persisted or shared state, or schema, and makes **no** new user-facing claim or capability; `tsc --noEmit`, `prettier --check .`, and the production build pass; you have **manually verified it on the dev server across the affected viewports and states** (no front-end test runner yet — [#173](https://github.com/alethical-org/alethical/issues/173) — so manual QA is the gate); and a clean `git revert` fully undoes it. Say in the PR what you verified. **Accessibility fixes verify by their standard, not by eye, and are not held for approval:** file the issue at discovery, fix, then gate on `tsc`/`prettier`/CI/build green **plus** a WCAG check (screen-reader name/heading, reduced-motion toggle, a measured contrast ratio) — for an already-shipped public surface, live confirmation may follow the merge; when an AA fix darkens a shared token or carries another visual tradeoff, still merge but name the tradeoff in the after-report so it can be eyeballed and reverted if disliked. **Local routing wiring** qualifies only if you've confirmed every affected route — and any shareable/deep-link URL (`.claude/rules/grounded-answers.md` rule 5) — still resolves. Still propose-first: data / auth / persisted-state / schema / migrations, new-capability or claim-making copy (`.claude/rules/grounded-answers.md` + copy-claims invariants), outward-facing / published content, and anything you can't visually verify (e.g. an auth-gated surface). The governing test is **blast radius + reversibility + verifiability** — not whether it's "just styling" or "production code." Bias toward acting; keep widening this carve-out as new safe patterns prove out (propose the widening here in the same session), and propose narrowing it if something bites.

   **Safe, verified backend changes are auto-eligible too (granted Jul 14 2026).** A backend / pipeline / API change qualifies for build-and-squash-merge when **all** of these hold: it is behavior-preserving by default or its new behavior is fully exercised by tests (e.g. a parameterization whose defaults keep current behavior, a bug fix with a regression test); the full backend suite passes **locally against real Postgres** and the **backend CI job is green** on the PR; a clean `git revert` undoes it (no schema/data migration, no destructive or irreversible state change); and it spends no money and touches no production *data* (writing code that will later run against production is fine — *running* it against production is not). Reference precedents: [#219](https://github.com/alethical-org/alethical/pull/219) (session parameterization, defaults unchanged) and [#222](https://github.com/alethical-org/alethical/pull/222) (fallback-embedding labeling fix with regression test + real-data proof) — both CI-green and merged under this clause's grant. Say in the PR what was verified and how. **Still higher-scrutiny (but not an automatic pause):** *destructive/irreversible* schema / data migrations (Alembic) — an **additive, reversible, verified** migration is auto-merge per the additive-migration clause above — anything auth/security-sensitive, destructive or hard-to-reverse data operations, *unverified* production data runs (a backfill you've dry-run-proven and made idempotent is auto-merge; verify by engineering, not by asking), new externally-visible API contracts or user-facing claims, and changes whose correctness you cannot demonstrate by test or local execution. The governing test stays blast radius + reversibility + verifiability; "it's backend code" alone is no longer a reason to hand the merge back.

   **Zero-behavior cleanup is always in-lane — correct it, don't offer.** A change that alters no runtime behavior and makes no new user-facing claim clears the carve-out *by construction*, so just do it and merge: fixing a stale, orphaned, or now-false code comment or reference that a change created (rule 6 fallout), a typo, a dead-comment tidy-up, a broken cross-link, or the like. This holds **even in app / front-end code, and even in code another session just shipped** — correct it to match current reality now; never defer on "it's their code," "the underlying value might change later," or "the user is wrapping up." Verify the cheap way (the diff itself for prose; `tsc` / `prettier` when the file also compiles), then PR and squash-merge in the same session. Catching yourself about to ask "want me to fix this comment / tidy this up?" *is* the propose-and-offer this rule forbids — the answer is already yes. (Still propose-first only if the "cleanup" quietly changes behavior, a claim, or a public contract — then it isn't zero-behavior and a different clause governs.)

   **A change you recommend is as authorized as one the user typed — don't hand the commit back.** The autonomy test is blast-radius + reversibility + verifiability, never *who thought of it*. The previous clause covers zero-behavior cleanup; this covers the rest: a change that *does* alter behavior but is still safe / reversible / verifiable and came from your own analysis — a follow-on you spotted, a **ripple from another session's merge** (Jul 2026: [#206](https://github.com/alethical-org/alethical/pull/206) shortened the capability cards' press-glow to 300ms, leaving the chips at 650ms — inconsistent with what [#204](https://github.com/alethical-org/alethical/pull/204) built, so aligning them ([#208](https://github.com/alethical-org/alethical/pull/208)) was mine to ship, not to hand back) — gets executed and reported after, exactly as if the user had asked. None of these turn it back into a "want me to ship it?": *it was my idea, not the user's* (the recommendation **is** the request — this makes binding, for the repo PR pipeline, `~/.claude/CLAUDE.md`'s "act on your recommendation, report after"), *it depends on inferring intent* (when the intent follows from the task's stated goal or the code, reason it out and act), or *another session chose the thing I'm rippling from deliberately* (deliberate-there ≠ off-limits-here). Reserve asking for the two genuine cases only: a real **toss-up** — no basis to prefer an option, so surface it briefly — and the **propose-first categories** (data / auth / persisted-state / schema / migrations, claim-making copy, outward-facing / published content, anything you can't verify). If the reason you're about to ask reduces to "it wasn't explicitly requested," that reason is void.

   **Applying `workflow-overhead-audit` verdicts is in this lane too.** When that audit produces keep/trim/merge/drop verdicts on non-exempt gates (rules, skills, docs, `.claude/` config), apply them in the same pass — including meaning-preserving rewrites of a rule or skill file. The audit record's before/after diff *is* the verification and `git revert` cleanly undoes it, so the "can't cheaply verify → propose-first" bar does not gate audit edits; don't stop to ask. Only the audit's own exempt safety rails stay off-limits (the leak-guard commit hook, `.claude/rules/grounded-answers.md` invariants, CI correctness/format checks). If you catch yourself about to ask "want me to merge this?" for a verified self-contained front-end change or an audit-verdict edit, the answer is already yes. The one thing that actually defeats this rule is reading a *stale* copy of it — a worktree branch or CLAUDE.md snapshot cut before a carve-out merged still shows the old "app code → propose first" text. Before deciding this rule doesn't cover your change, confirm you're on the current `origin/main` version (rule 1), not a stale checkout.

   **Execute the loop safely when sessions run concurrently.** The autonomous loop compresses build→merge into one unattended burst, and this repo runs many sessions and worktrees at once — origin, and sometimes your own working directory, move under you. Guard the merge, not just the start (rule 1):

   - **Default to your own worktree — don't build in the shared checkout.** Before you write code that you'll commit, create a dedicated worktree off `origin/main` and work there: `just worktree <branch>` (creates `../alethical-wt-<branch>`, installs deps, and links `.env` so build/verify works immediately — the node_modules friction that used to make isolation "too slow" is handled). `just worktree-rm <branch>` cleans it up after the PR merges. This is the *default starting move* for build/commit work, not a reaction to noticing a collision — isolation is what makes concurrent sessions safe. (The bare `git worktree add -b <branch> <path> origin/main` still works if you need a custom path; the recipe just removes the setup toil.)
   - **Never run destructive or branch-switching git on the shared checkout.** `git reset --hard`, `git checkout .` / `git checkout -- .`, `git clean -fd`, discarding `git stash`, and `git checkout <other-branch>` on a checkout other sessions may be using will **silently wipe their uncommitted work** and yank the branch out from under them — this actually happened (Jul 2026: a session's cleanup `reset`/`checkout` wiped two other sessions' in-progress edits). In your own worktree these ops are safe because they only touch your isolate. If you must touch the shared checkout, only ever *add* (never discard), and prefer `git worktree` over switching its branch.
   - **A push is the real safety net for uncommitted work.** If you can't isolate and must build in a shared checkout, commit only *your* files (`git add <paths>`, never `git add .` when foreign changes are present) and **push immediately** — a pushed commit survives any other session's destructive clean; uncommitted or unpushed work does not.
   - **Re-verify freshness at merge, against the fresh tip.** Right before committing and again before merging, `git fetch origin main` then `git diff origin/main -- <file>` — trust that, not your in-memory read: a file can mutate between your read and your edit when a PR merges mid-session (this session, Jul 2026: #200 rewrote a skill I had open; #203 landed while working). Confirm the diff is *only* your hunks — a tangled or duplicated diff means another session already shipped part of it (rule 1's #121/#122 trap, caught at merge instead of after).
   - **"Skipped" is not "verified."** A CLEAN merge state with the relevant CI job SKIPPED (path-filtered — e.g. the front-end job on a docs-only PR, so repo-wide `prettier --check .` never ran) means that gate didn't fire. You are the last gate in this lane: run the skipped check locally, or consciously accept its absence and say so in the PR.
   - **Finish cleanup by hand when a worktree holds `main`.** `gh pr merge --delete-branch` can't check out `main` or delete your local branch when another worktree holds them — its `failed to run git` prints *after* the remote squash lands, not instead of it. Confirm the squash is on the `origin/main` tip, then clean up by hand (`git branch -D <branch>`, `git worktree remove <path>`); don't read the error as a failed merge.

   **Take the lead on product management and development — proactive sweeps and dead-code removal included (Eugene, Jul 16 2026).** The maintainer put you in the lead on this platform's product management and engineering: don't wait to be pointed at each piece of work, and don't checkpoint before safe/reversible/verifiable work just because it wasn't spelled out. Leading the product means driving the roadmap — identifying, prioritizing, and sequencing the work (`Own the sequencing` above) and carrying it to done (`Prioritize continuous, uninterrupted progress` above), not asking which thing to do next. Two behaviors this makes standing, both **without a checkpoint**:

   - **Sweep the pattern — don't fix only the reported instance.** When a request exposes a *class* of problem — a redundancy, a bug shape, a stale reference, an inconsistency, an accessibility miss, a naming drift — search the whole codebase for every instance in the same pass, fix every clear-cut one, and ship them together. Present back only the genuine judgment calls (a real toss-up, or wording/behavior a reasonable person would weigh differently), and for those lead with your recommendation, never a bare menu. Fixing the one instance the user named and handing back the rest you found *is* the disguised handback rules 10–11 forbid. (Origin: asked to drop one redundant empty-state line, I fixed it but handed back the follow-up sweep and the dead-screen removal instead of doing them.)

   - **Solve the broader problem when you're already there — don't narrow-and-defer (Eugene, Jul 22 2026).** The sibling of "sweep the pattern": that clause says fix every *instance* you find; this one says build the *more complete solution* when the work you're already doing exposes it, it addresses the same root problem the task named, and it is itself in-lane (safe / reversible / verifiable — an additive migration, a backend query change, a front-end reework). Do **not** ship the easy half, file the hard half as a follow-up issue, and stop — that is a disguised handback, and filing an issue does not discharge it. The trigger to build-now (not file-and-defer) is: you're in the exact context, the broader fix clears the same autonomy test as the narrow one, and it makes the feature actually deliver what the task was reaching for. Filing an issue is correct only for work that is genuinely *out* of this lane — a real toss-up on approach, a propose-first category, or a dependency you don't control — and even then you file it *and keep going* on what you can do. When a broader fix needs one genuine empirical/verification input (e.g. tuning a threshold against production data), that input is something you **go get** (measure it), not a reason to defer the whole build; ship it behind the superset/safety guarantee that makes a wrong tuning harmless, then refine. Milestone is irrelevant here too (rule 13): "the broader fix is v2" is never a reason to defer it when you're already positioned to build it now. (Origin: shipping #571's root-word search fix, I scoped down to inflectional matching, filed the broader fuzzy/typo/relevance fix as #573, and stopped — when #573 is an additive-migration + backend change in the auto-merge lane and I was already in the search code with a QA environment running. The narrow fix left the user's own reported goal — "near-matches surface" — only half-delivered.)

   - **Removing genuinely-unreachable code is in-lane** — same blast-radius + reversibility + verifiability test as every clause above. Dead code (an unmounted screen, an unrouted component, an unreferenced export/asset/style) is safe to delete: `git revert` restores it and nothing user-facing changes. The gate is **proof of unreachability, not permission** — establish it by *knowing*, not inferring (nothing routes/links/deep-links to it, or its navigator/entry is itself unrendered; grep the wiring and cite the evidence), then remove it with the imports/registration your deletion orphans, verify (`tsc`/`prettier`/build green, and the live app still routes everything that *is* reachable), and merge. If reachability stays genuinely uncertain *after* real investigation, that uncertainty is the signal to leave it (or trim rather than delete) — not to hand it back unexamined.

   The one hard line is unchanged: *unverified irreversible real-world harm* (a real send to real users, spending real money on an unverified run, a destructive/irreversible production data / schema / access change), honored by engineering — build behind config, dry-run, scoped live check — not by asking. Everything short of that, lead and act, then report.

   **Unattended / overnight: context pressure and task size are checkpoint-and-continue signals, never stop signals (Eugene, Jul 21 2026).** When the maintainer leaves work running unattended (overnight, "work while I sleep", "keep going"), the *point* is that the extra hours produce shipped work — so keep taking the lead through the whole safe/verifiable backlog until it's genuinely done or genuinely gated. **These are NOT reasons to stop early and hand back a tidy summary:** "my context is getting full," "it's a big / backend change," "it's late in the session," "it wasn't explicitly requested," "this is better as a fresh session." Each is a signal to **checkpoint and continue**, not to quit: keep the resumable in-repo plan current (rule: "Optimize for context"), commit at milestones, **offload well-specified sub-tasks to subagents** (each has its own context window) and parallelize independent ones, and work economically — then push the next item to done. Filing a safe, in-lane item as an issue and stopping, when you had the environment and the lane to finish it, is the disguised handback this forbids. (Origin, Jul 21 2026: after shipping the web Bill Detail overnight I filed the version-label polish [#433] and the per-member-votes backend [#83] and stopped — #433 was pure in-lane frontend polish I had a running prod-QA env to finish, and #83 fit the backend auto-merge lane; I treated "low context / big change / fresh session" as stops when they were checkpoint-and-continue signals.) The reserved stops are unchanged and are the *only* ones: (1) *unverified irreversible real-world harm* (the hard line above); (2) a genuine **toss-up** with no basis to prefer; (3) a genuine **correctness / verification gate you cannot establish** — e.g. a grounded-answers data-quality gate like #83's name-match coverage spike. Note that (3) means **investigate the gate** (measure the coverage, read the source, sample the data) and act on what you find — ship if it clears, document precisely if it doesn't — *not* "stop because a gate might exist." Issues you file for a genuinely-gated item should say what was investigated and what specifically blocks it, so a fresh session (or you, resumed) starts from the finding, not from scratch.

11. **Lead issues, PRs, and substantive comments with a plain-language "Net:" line.** Every GitHub issue, PR body, and substantive PR/issue comment opens with (or clearly labels near the top) a one–two-sentence **Net:** — the bottom line in plain words a non-specialist maintainer can act on: *what it means / what to do / why it matters*, not the mechanism. The full technical detail still follows in the body; the Net line is the layer that makes the takeaway graspable at a glance without parsing the internals. Same convention as the "Net:" line on decision options and chat wrap-ups (`~/.claude/CLAUDE.md`; memory `options-include-effort-and-reasoning`). A reader who only reads the Net line should still come away knowing the point and the ask.

   **Two levels, not one.** Beyond the overview Net at the top, **every acceptance-criteria (or checklist) item carries its own inline `— Net:` gloss** next to its technical wording — the plain-language *what this means / why it's checked* for that specific item — so the list is legible to a non-specialist line by line, not only in aggregate. Keep each item's gloss to a short clause; the technical criterion stays (it's what's verified), the Net just complements it. When editing an existing issue/PR that predates this, retrofit both levels. This applies to the issues and PRs *you* author or own; don't rewrite another session's merged artifact — add a Net comment there instead.

12. **Expensive background research is checkpoint-and-resume — never let it run into paid overage (Eugene, Jul 18 2026).** Multi-agent research runs (deep-research and similar Workflow fan-outs) are the costliest thing a session does, and **any of the user's usage limits can run out mid-run — the 5-hour session window, the weekly all-models allowance, or the weekly per-model (e.g. Fable) allowance. This rule applies to all three meters equally.** A weekly meter can force a multi-day pause, so the checkpoint must be durable enough to survive it. Standing practice:

   - **Checkpoint at launch.** When launching a long research workflow, immediately save a checkpoint file (scratchpad or session dir) with the run ID, script path, and the *exact* `args` string, so a later turn — possibly after context summarization — can resume without reconstructing anything.
   - **Stop before the limit, not after.** If the user pauses for cost or any of the three meters is near its limit, stop the run (`TaskStop`) rather than letting it burn into overage. This is safe: every completed agent is journaled (`journal.jsonl` in the workflow transcript dir) and replays from cache at zero cost; a stop loses only in-flight agents.
   - **Resume from cache, byte-identical.** Resume with `Workflow({scriptPath, resumeFromRunId, args})` where `args` is byte-identical to the original — *omitting* `args` aborts instantly without touching the cache (Jul 2026: a resume without args died with "No research question provided"), and *changed* args miss the cache and re-pay for everything. Same-session only.
   - **Run the tail cheap.** The resumed remainder (final verifies, synthesis) inherits the session model — switch to the cheapest model adequate for those stages before resuming.
   - **Never re-run whole when a journal exists.** Before diagnosing an empty or failed result as lost work, read the journal — the cached results are usually all there.
   - **Never promise a threshold stop — pick a mechanism before ending the turn.** Claude cannot see any of the usage meters and does not run between turns, so "I'll stop it at N%" is unenforceable (Jul 18 2026: promised a 97% stop with no mechanism; the user had to stop the run manually at 98%). When the user names a threshold on any meter or reports one running hot, do one of two things before ending the turn — and say which one was done: `TaskStop` the run immediately (the journal cache makes stopping cheap), or arm a timed one-shot stop sized to the remaining tail (`send_later`, e.g. 15–20 minutes out: on fire, stop the run if it's still going and re-checkpoint). A stated intention with no armed mechanism is the failure mode, not a plan. Model choice is a lever here too: the weekly per-model meter drains only for that model's usage, so running research tails on a cheaper model spares the premium meter for judgment work.

13. **Staff every goal-supporting issue — delegate the ones you're not doing yourself; milestone never gates this (Eugene, Jul 22 2026).** When a goal touches more than one outstanding issue, *every* issue that supports reaching that goal must have an **active owner** — this session, or another session you've prompted — never sitting unstaffed in the backlog. As you work a goal, enumerate the issues it depends on or spins off (including the durable/source fixes behind any interim patch you shipped, and the honest fix behind any design item you declined), and for each one either **work it here**, **confirm a running session already owns it**, or **prompt a fresh session to take it** — `spawn_task` a self-contained, direct-address brief (or `send_message` an existing session), telling it to coordinate back via the issue's comments, exactly as done for #483 / #520 / #550 / #551. Leaving a goal-supporting issue with no owner — "filed, parked, someone will get to it" — is the failure this prevents; the maintainer put you in the lead, and coverage is the lead's job. After delegating, **monitor** the sessions and **finish any dependent integration** yourself.

    **Always tell a delegated session to `send_message` you back the moment it finishes (Eugene, Jul 22 2026).** Every delegation brief (`spawn_task` / `send_message`) must end with an explicit instruction: *"when you're done — PR merged or work blocked — `send_message` this session (name + id) with the outcome (PR #, merged/blocked, anything I need to integrate)."* Do NOT rely on the background task-completion notification to learn a delegated issue shipped: that fires only when the session's *process* ends, which lags — often long after the PR merged, sometimes never within your turn — so a merged issue can sit uncaught until you happen to poll GitHub (Jul 22 2026: #520 merged via #567 and I only noticed on a manual sweep, because I'd received the "started" ping but no timely "finished" one). The direct message arrives in-thread as a user turn, so you can immediately confirm the work, run dependent integration, or restaff on a block — instead of discovering it late. Include this session's own identity in the brief so they know who to message.

    **Milestone is irrelevant to staffing — `v1`/`v2`/`v#` mean nothing for whether work gets done now (Eugene, Jul 22 2026).** Milestones only *group and report* work; they are **not** a defer signal. Never leave a goal-supporting issue unstaffed, or decline to pull it forward, *because* it sits in a later-numbered milestone — if it supports the goal at hand, staff it now regardless of its bucket, and re-milestone it if the bucket no longer reflects reality. This **overrides** any earlier phrasing in these rules that reads a higher-numbered milestone as "deferred / do later" (e.g. rule 4's phase language and the "defer by moving to a higher bucket" note): those describe how milestones *label* phase, not a license to leave goal work unowned. When you catch yourself about to park a supporting issue as "it's v2," that thought is the trigger to instead give it an owner.
